# Open-source Project Shape

## Why open source this

Prly touches a problem many people feel but very few tools solve cleanly: how to turn fragmented personal work signals into a reliable execution layer.

Open source is useful here because:

- source adapters vary across user environments
- safety and idempotency benefit from public scrutiny
- schemas and contracts are more valuable when shared
- local-first operation matters for trust and privacy

## Near-term shape

The project starts with:

- docs-first product framing
- shared Python schemas and tests
- GitHub-native issue and roadmap structure
- CI for repository hygiene and model evolution

## Future expansion candidates

- pluggable collectors
- typed artifact storage formats
- action policy engine
- local dashboard or TUI
- memory fan-out adapters
